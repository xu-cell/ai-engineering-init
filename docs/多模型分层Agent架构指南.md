# 多模型分层 Agent 架构指南

> 按思维深度分层：Haiku 搬数据、Sonnet+Codex 分析代码、Opus 需求分析+写代码。各司其职，快省兼得。

---

## 架构总览

```
┌─────────────────────────────────────────────────────────┐
│            Haiku 层 — "数据搬运工"                       │
│            快、省、并行跑、只搬不想                       │
│                                                         │
│  code-scanner     代码库扫描 + 文件定位 + 代码片段收集   │
│  loki-runner      Loki 日志查询 + 格式化                 │
│  mysql-runner     MySQL 只读查询 + 格式化                │
│  task-fetcher     云效任务获取 + 整理                    │
│  image-reader     截图/图片/Axure原型 内容提取           │
└────────────────────┬────────────────────────────────────┘
                     │ 结构化数据
                     ▼
┌─────────────────────────────────────────────────────────┐
│            Sonnet + Codex 层 — "代码分析师"               │
│            理解代码逻辑、分析根因、审查规范               │
│                                                         │
│  bug-analyzer     Sonnet 理解上下文 + Codex 逻辑分析    │
│  code-reviewer    Sonnet 规范检查 + Codex 逻辑审查      │

                     │ 分析结论 / 审查报告
                     ▼
┌─────────────────────────────────────────────────────────┐
│            Opus 层 — "架构师 + 需求分析 + 编码"           │
│            需求分析、架构决策、编写代码、协调全局         │
│                                                         │
│  requirements-analyzer  协调 Haiku 层，输出需求分析报告  │
│  主会话                 架构决策 + 编写核心代码           │
└─────────────────────────────────────────────────────────┘
```

## 各 Agent 详细说明

### Haiku 层（4 个 Agent）

#### loki-runner

| 属性 | 值 |
|------|-----|
| 模型 | haiku |
| 职责 | 查询 Loki 日志，格式化输出 |
| 触发 | Bug 排查、线上问题、traceId 追踪 |
| 输入 | traceId / 接口路径 / 错误关键词 / 环境名 |
| 输出 | 日志条目 + 错误类型 + 发生位置 + 租户ID |

**示例调用**：
```
"查 test13 环境 traceId=abc123 的日志"
→ loki-runner(Haiku) → 3秒返回格式化日志
```

#### mysql-runner

| 属性 | 值 |
|------|-----|
| 模型 | haiku |
| 职责 | 执行 MySQL 只读查询，格式化结果 |
| 触发 | 数据排查、验证数据状态 |
| 输入 | SQL / 表名 / 数据库名 / 租户ID |
| 输出 | 查询结果表格 + 行数 |

**安全约束**：只允许 SELECT / SHOW / DESCRIBE，禁止任何写操作。

**示例调用**：
```
"查 tenant_123 库的 order_info 表最近 10 条失败订单"
→ mysql-runner(Haiku) → 2秒返回数据表格
```

#### task-fetcher

| 属性 | 值 |
|------|-----|
| 模型 | haiku |
| 职责 | 获取云效任务，整理为结构化清单 |
| 触发 | 需求同步、任务查询、工作项读取 |
| 输入 | 任务编号 / 项目名 / 查询条件 |
| 输出 | 任务列表表格 / 任务详情 |

**示例调用**：
```
"查一下 SARW-456 的任务详情"
→ task-fetcher(Haiku) → 2秒返回任务信息
```

#### image-reader

| 属性 | 值 |
|------|-----|
| 模型 | haiku |
| 职责 | 读取图片/截图/Axure原型图，提取结构化信息 |
| 触发 | 错误截图、表格截图、架构图、Axure 原型分析 |
| 输入 | 图片文件路径 |
| 输出 | 提取的文字 + 结构描述 / Axure 页面结构 + 字段列表 + 交互说明 |

**Axure 原型分析能力**：自动识别原型图并提取页面结构、搜索条件、表格列、表单字段、操作按钮、交互流程，输出结构化开发需求清单。

**注意**：复杂架构图如果 Haiku 提取不够精确，Opus 主会话可以直接用自身视觉能力补充分析。

### Sonnet + Codex 层（2 个 Agent）

#### bug-analyzer

| 属性 | 值 |
|------|-----|
| 模型 | sonnet |
| 职责 | Bug 根因分析，结合 Codex 做代码逻辑分析 |
| 触发 | 接收 Haiku 层数据后需要分析根因 |
| 输入 | 日志数据 + 数据库数据 + 代码片段 |
| 输出 | 根因分析报告 + 修复建议 + 验证方法 |

**双重分析**：
1. Sonnet 自身：理解错误上下文、定位代码、分析业务逻辑
2. Codex MCP：分析代码逻辑缺陷（空指针、并发安全、SQL 注入等）

#### code-reviewer

| 属性 | 值 |
|------|-----|
| 模型 | sonnet |
| 职责 | 双重代码审查（规范 + 逻辑） |
| 触发 | /dev、/crud 完成后，或用户说"审查代码"、"review" |
| 输入 | git diff 变更文件 |
| 输出 | 审查报告（致命问题 + 重要问题 + 通过项 + Codex 逻辑意见） |

**双重审查**：
1. Sonnet 自身：项目规范检查（包名、审计字段、权限注解、请求封装等 10 项）
2. Codex MCP：代码逻辑审查（空指针、事务一致性、并发安全等）

### Opus 层（1 个 Agent + 主会话）

#### requirements-analyzer

| 属性 | 值 |
|------|-----|
| 模型 | opus |
| 职责 | 需求分析，协调 Haiku 层获取数据，输出开发任务清单 |
| 触发 | Axure 原型分析、产品需求转开发任务、需求拆解 |
| 输入 | 原型截图 / 云效任务编号 / 需求描述 |
| 输出 | 需求分析报告（数据库设计 + 接口清单 + 任务拆解 + 推荐流程） |

**核心能力**：
1. 协调调用 image-reader(Haiku) 提取原型图结构
2. 协调调用 task-fetcher(Haiku) 获取云效任务信息
3. 基于结构化数据进行需求分析和任务拆解
4. 输出符合项目规范的数据库设计、接口设计、开发计划

**示例调用**：
```
"分析这几张 Axure 原型截图，给出开发任务"
→ requirements-analyzer(Opus)
  ├─ image-reader(Haiku) × N张 → 页面结构数据     [并行，3秒]
  └─ 需求分析 + 任务拆解                          [10秒]
→ 输出: 需求分析报告 + 开发任务清单
```

## 典型工作流

### 场景 1：需求分析 → 开发（analyze-requirements 技能编排）

> 由 `analyze-requirements` 技能编排全流程。核心原则：简单需求直接分析，复杂需求走 Agent。

```
用户: "分析这几张原型截图，开发这个功能 [图片1] [图片2] [图片3]"

Step 0: Opus 主会话 → 复杂度判断
  │
  ├─ 简单需求？ → 直接分析，输出任务清单 → 推荐 /crud 或 /dev
  │   例：纯文字描述、单表 CRUD、字段列表已明确
  │
  └─ 复杂需求？ → 进入 Agent 路径 ↓

Step 1: Opus 主会话 → 启动 requirements-analyzer(Opus)
  │
  ├─ 按信息量决定内部调用组合：
  │   | 用户提供的信息          | 内部启动的 Haiku Agent             |
  │   |------------------------|------------------------------------|
  │   | 只有文字描述            | 不启动（快速路径）                  |
  │   | 文字 + 原型截图         | image-reader × N张                 |
  │   | 文字 + 云效任务号       | task-fetcher                       |
  │   | 原型截图 + 云效任务号   | image-reader + task-fetcher（并行） |
  │
  ├─ 内部并行调用 Haiku 层
  │   ├─ image-reader(Haiku) → 分析图片1            [3秒]
  │   ├─ image-reader(Haiku) → 分析图片2            [3秒]
  │   └─ task-fetcher(Haiku) → 获取任务详情          [2秒]（如有）
  │
  ├─ 汇总结构化数据
  │   ├─ 识别业务实体 → 推导数据库表设计
  │   ├─ 梳理页面关系 → 确定接口清单
  │   └─ 评估复杂度 → 拆解开发任务
  │
  └─ 输出: 需求分析报告                             [10秒]
     · 数据库设计（建表 SQL）
     · 接口清单（路由 + 方法）
     · 开发任务清单（按依赖排序）
     · 推荐开发流程（/crud or /dev or OpenSpec）

Step 2: Opus 主会话 → 按任务清单逐步开发
  ├─ /crud 生成基础 CRUD                             [Opus 编码]
  └─ /dev 补充业务逻辑                               [Opus 编码]

Step 3: Opus 主会话 → 启动审查
  └─ code-reviewer(Sonnet+Codex) → 审查代码          [8秒]
     返回: 审查报告（通过/需修改）

Step 4: 完成
```

### 场景 2：云效任务 + 原型图 → 完整开发

```
用户: "看看云效 SARW-456 的需求，原型截图在这 [图片]"

与场景 1 相同流程，analyze-requirements 技能自动判断：
- 有原型截图 → image-reader(Haiku)
- 有云效任务号 → task-fetcher(Haiku)
- 两者并行启动，汇总后输出需求报告

Step 1: requirements-analyzer(Opus)
  ├─ task-fetcher(Haiku) → 获取 SARW-456 详情        [2秒]
  └─ image-reader(Haiku) → 解析原型截图              [3秒]
  └─ 输出: 需求分析报告 + 开发任务清单               [10秒]

Step 2: Opus 主会话 → 按任务清单开发                 [Opus 编码]

Step 3: code-reviewer(Sonnet+Codex) → 审查           [8秒]
```

### 场景 3：Bug 修复（fix-bug 技能编排）

> 由 `fix-bug` 技能编排全流程。两阶段 Agent 排查 + 报告确认 + 修复提交。

```
用户: "线上订单创建报 500，traceId=abc123"

Step 0: Opus 主会话 → 复杂度判断
  │
  ├─ 简单 Bug？ → 读代码定位 → 输出排查报告 → 用户确认 → 修复 → git-workflow
  │   例：明显 NPE、拼写错误、用户已指出修改方案
  │
  └─ 复杂 Bug？ → 进入两阶段 Agent 路径 ↓

Step 1（阶段一）: Opus 主会话 → 按信息量并行启动 Agent（必须同时启动）
  ├─ bug-analyzer(Sonnet+Codex)  → 读代码分析根因        [10秒]（必启动）
  ├─ loki-runner(Haiku)          → 查 traceId 日志        [3秒]（有 traceId 时）
  └─ mysql-runner(Haiku)         → 查数据库验证数据       [2秒]（有 DB 信息时）

  按信息量决定启动组合：
  | 用户提供的信息          | 阶段一启动                                |
  |------------------------|-----------------------------------------|
  | 只有 Bug 描述           | bug-analyzer                             |
  | Bug 描述 + traceId      | bug-analyzer + loki-runner               |
  | Bug 描述 + DB 信息      | bug-analyzer + mysql-runner              |
  | Bug 描述 + traceId + DB | bug-analyzer + loki-runner + mysql-runner |

Step 2（阶段二）: 日志驱动二次查询（阶段一返回后）
  从 loki-runner 结果中提取表名/数据ID/租户ID
  ├─ 有可查询信息？ → 启动 mysql-runner 二次查询验证数据
  └─ 无数据库线索？ → 跳过，直接汇总

Step 3: Opus 主会话 → 汇总所有结果 → ⚠️ 输出排查报告（不修改代码）

Step 4: ⏸️ 等待用户确认修复方案（用户说"修"或确认后继续）

Step 5: Opus 主会话 → 编写修复代码                        [Opus 编码]

Step 6: Opus 主会话 → 启动审查
  └─ code-reviewer(Sonnet+Codex) → 审查修复代码           [8秒]
     返回: 审查报告（通过/需修改）

Step 7: git-workflow 提交（必须通过 git-workflow 技能，禁止直接 git commit）
```

#### fix-bug 与 bug-detective 的区别

| 维度 | fix-bug | bug-detective |
|------|---------|---------------|
| 定位 | **全流程编排**（排查+修复+提交） | 排查指南（方法论+决策树） |
| Agent 使用 | 自动编排并行 Agent | 不涉及 Agent |
| 输出 | 修复代码 + git 提交 | 排查方向建议 |
| 触发 | "修复 bug"、"fix bug" | "排查"、"报错"、"异常" |

## 设计原则

1. **按思维深度分层**：搬数据不需要强推理，分析代码需要中等推理，需求分析和写代码需要强推理
2. **Haiku 只搬不想**：严格限制 Haiku agent 只做数据获取和格式化，不做分析
3. **Sonnet 分析不写**：Sonnet 层只输出报告和建议，不直接修改代码
4. **Opus 分析+决策+编码**：需求分析、架构决策、代码修改都在 Opus 层完成
5. **Agent 可嵌套调用**：Opus 层 agent 可以内部调用 Haiku 层 agent，实现层级协作
6. **Codex 补充逻辑**：Sonnet 层可调用 Codex MCP 做补充的代码逻辑分析
7. **后台并行**：Haiku 层 agent 尽量后台并行运行，不阻塞主流程
