## 1. 问题与动机

### 当前痛点

```
用户："帮我在订单模块加一个退款功能"

当前流程（Opus 主会话亲自扫描，每次 20+ 次工具调用）：
  Opus → Glob("**/order/**/*.java")           # 找文件
  Opus → Grep("class.*Order.*Controller")     # 找 Controller
  Opus → Read(OrderInfoWebController.java)    # 读 Controller
  Opus → Grep("class.*Order.*Business")       # 找 Business
  Opus → Read(OrderWebBusiness.java)          # 读 Business
  Opus → Grep("class.*Order.*Service")        # 找 Service
  Opus → Read(OrderInfoService.java)          # 读 Service
  Opus → Read(OrderInfo.java)                 # 读 Entity
  Opus → Read(OrderInfoMapper.java)           # 读 Mapper
  Opus → Grep("order_info")                   # 找表结构
  ... 还有 VO、DTO、枚举、配置 ...
  ────────────────────────────────
  总计：15-25 次工具调用，消耗大量 Opus token
  耗时：30-60 秒的扫描阶段
```

### 目标流程

```
预先构建（一次性，Haiku 执行）：
  npx ai-engineering-init index --build
  └─ code-scanner(Haiku) × N 模块并行 → .ai-index/*.md

任务执行（每次，Opus 消费）：
  Opus → Read(.ai-index/MANIFEST.md)          # 1 次，了解全局
  Opus → Read(.ai-index/modules/sys-canteen.md) # 1 次，了解目标模块
  Opus → 开始编码                               # 直接开始
  ────────────────────────────────
  总计：2-3 次 Read，节省 80%+ 扫描 token
  耗时：2-3 秒

  如果索引信息不够 → 降级到 code-scanner(Haiku) 实时补充
```

### 量化收益预估

| 指标 | 当前（无索引） | 目标（有索引） | 节省 |
|------|--------------|--------------|------|
| 首轮工具调用 | 15-25 次 | 2-3 次 | ~85% |
| 首轮 token 消耗 | ~8000 tokens | ~1500 tokens | ~80% |
| 首轮耗时 | 30-60s | 2-5s | ~90% |
| 模型层级 | Opus 执行扫描 | Haiku 预构建 | 成本降 90%+ |

## 2. 核心设计理念

### 2.1 三个核心原则

```
原则 1：Markdown 即索引
  索引文件就是 .md，AI 用 Read 直接读取，无需解析器
  像 Skill 一样被消费，无需额外工具链

原则 2：先粗后细、按需加载
  MANIFEST（500 tokens）→ 模块卡片（1000 tokens）→ 符号索引（按需）
  Opus 只读需要的层级，不会一次加载全部

原则 3：增量更新、防漂移
  基于 git diff 只重建变更涉及的分片
  定期全量校验防止索引与代码不一致
```

### 2.2 与 Skill 的互补关系

```
Skill  = 教 AI "怎么写"（规范、模式、禁止项）  → 很少变化，手动维护
Index  = 告诉 AI "代码在哪"（结构、文件、符号） → 频繁变化，自动生成

两者协同：
  1. Opus 读 Skill 获得开发规范
  2. Opus 读 Index 获得代码上下文
  3. Opus 开始编码（两手信息都有）
```

### 2.3 设计约束

| 约束 | 说明 |
|------|------|
| **零外部依赖** | 纯 Node.js 实现，不依赖 tree-sitter 等 native 库 |
| **跨语言通用** | Java/Spring、Vue/React、Node.js 都能索引 |
| **存储为 Markdown** | 不用 JSON/SQLite，保证 AI 直接可读 |
| **单分片 < 1500 tokens** | 防止上下文窗口爆炸 |
| **git 友好** | 索引文件可 commit，也可 .gitignore |

## 3. 索引目录结构

### 3.1 完整目录树

```
.ai-index/
│
├── MANIFEST.md                      # L0：全局入口（必读）
│                                    #   项目指纹、模块列表、技术栈、索引元信息
│                                    #   目标：< 500 tokens
│
├── modules/                         # L1：模块卡片（按任务读 1-2 个）
│   ├── {module-name}.md             #   模块职责、入口文件、核心类、依赖、数据表
│   └── ...                          #   目标：每个 < 1000 tokens
│
├── capabilities/                    # L2：能力索引（按需读取）
│   ├── api-routes.md                #   URL → Controller → 方法签名
│   ├── data-models.md               #   表名 → Entity → VO/DTO 映射
│   ├── jobs-events.md               #   定时任务、MQ 消费者、事件监听
│   └── enums-constants.md           #   枚举类、常量定义
│
├── symbols/                         # L3：符号索引（精确查找时读取）
│   ├── symbols-{shard}.md           #   类名/方法名 → 文件:行号
│   └── ...                          #   按首字母或模块分片
│
├── files/                           # L4：文件卡片（深度了解时读取）
│   ├── files-{module}.md            #   路径 + 一句话摘要 + 导出符号 + 行数
│   └── ...
│
├── changes/                         # 变更追踪
│   ├── latest-delta.md              #   最近一次增量更新的变更摘要
│   └── history-{YYYY-MM}.md         #   月度变更汇总
│
└── state.md                         # 索引元信息
                                     #   git hash、构建时间、索引版本、各分片 hash
```

### 3.2 加载层级策略

```
任务开始时的加载决策：

  L0: MANIFEST.md          ← 始终读取（了解全局）
        ↓ 判断任务涉及哪个模块
  L1: modules/{module}.md  ← 按需读取 1-2 个模块卡片
        ↓ 信息是否足够？
  足够 → 直接开始编码
  不够 ↓
  L2: capabilities/*.md    ← 读取相关能力索引（API/数据模型）
        ↓ 还需要精确定位？
  L3: symbols/*.md         ← 查找具体符号位置
        ↓ 需要详细了解？
  L4: files/*.md           ← 读取文件卡片
        ↓ 索引中没有？
  降级: code-scanner(Haiku) ← 实时扫描补充
```

## 5. 构建算法

### 5.1 全量构建流程

```
输入: 项目根目录
输出: .ai-index/ 完整索引

Step 1: 项目识别
  ├─ 检测语言（Java/TS/JS/Vue/Python/Go）
  ├─ 检测框架（Spring Boot/Vue/React/Express）
  ├─ 检测包管理器（Maven/Gradle/npm/pnpm）
  └─ 检测架构模式（单模块/多模块/Monorepo）

Step 2: 模块发现
  ├─ Java: 扫描 pom.xml / build.gradle 的 modules
  ├─ Node: 扫描 workspaces / lerna.json
  ├─ 单模块: 按 src/ 下的顶级目录划分
  └─ 输出: 模块列表 [{name, path, type}]

Step 3: 并行扫描（每个模块一个 Haiku Agent）
  ├─ 文件树扫描（Glob）
  │   └─ 排除: node_modules, target, .git, build, dist
  ├─ 类/函数提取（Grep + 正则）
  │   ├─ Java: class/interface/enum 定义、@RestController、@Service
  │   ├─ TS/JS: export class/function/const
  │   └─ Vue: <script> 中的 export default
  ├─ 路由提取（Grep）
  │   ├─ Java: @RequestMapping/@GetMapping/@PostMapping
  │   ├─ Express: router.get/post/put/delete
  │   └─ Vue Router: path 定义
  ├─ 数据模型提取（Grep）
  │   ├─ Java: @TableName/@Entity
  │   ├─ SQL: CREATE TABLE
  │   └─ TypeORM/Sequelize: @Entity/@Table
  └─ 依赖关系提取
      └─ import/require 语句分析

Step 4: 聚合生成
  ├─ 生成 modules/{name}.md（每个模块）
  ├─ 生成 capabilities/*.md（聚合所有模块）
  ├─ 生成 symbols/*.md（聚合所有符号）
  ├─ 生成 files/*.md（聚合所有文件卡片）
  ├─ 生成 MANIFEST.md（全局入口）
  └─ 生成 state.md（索引元信息）

Step 5: 校验
  ├─ 检查每个分片 token 数 < 1500
  ├─ 超出的分片自动拆分
  └─ 计算覆盖率
```

### 5.2 语言适配器设计

```javascript
// 适配器接口
interface LanguageAdapter {
  // 语言标识
  id: string;                    // 'java', 'typescript', 'vue', etc.

  // 文件匹配
  filePatterns: string[];        // ['**/*.java'], ['**/*.ts', '**/*.tsx']

  // 排除目录
  excludeDirs: string[];         // ['target', 'build'], ['node_modules', 'dist']

  // 提取规则（正则）
  extractors: {
    classes: RegExp[];           // 类定义
    functions: RegExp[];         // 函数/方法定义
    routes: RegExp[];            // 路由定义
    models: RegExp[];            // 数据模型定义
    enums: RegExp[];             // 枚举/常量定义
    imports: RegExp[];           // 导入语句
  };

  // 模块发现
  discoverModules(rootDir: string): Module[];
}
```

```javascript
const JavaAdapter = {

```

#### Vue/JavaScript 适配器

```javascript
const VueAdapter = {
  id: 'vue',
  filePatterns: ['**/*.vue', '**/*.js', '**/*.ts'],

    models: [],

    ],
    imports: [
      /import\s+.*from\s+['"]([^'"]+)['"]/gm
    ]
  }
};
```

## 6. 增量更新机制

### 6.1 变更检测

```
增量更新触发条件：
  1. 手动执行 `npx ai-engineering-init index --update`
  2. Hook 触发（git commit 后 / 会话开始时）
  3. 定时任务（每天凌晨全量校验）

变更检测方法：
  优先: git diff --name-only {last-indexed-hash}..HEAD
  降级: 文件 mtime 对比（非 git 项目）
```

### 6.2 增量更新算法

```pseudo
function incrementalUpdate(repoRoot):
  # 1. 读取上次索引状态
  lastHash = readState().gitHash
  currentHash = gitRevParse("HEAD")

  if lastHash == currentHash:
    return "索引已是最新"

  # 2. 获取变更文件列表
  changedFiles = gitDiff(lastHash, currentHash, "--name-only")
  deletedFiles = gitDiff(lastHash, currentHash, "--diff-filter=D")
  addedFiles   = gitDiff(lastHash, currentHash, "--diff-filter=A")

  # 3. 确定受影响的模块
  affectedModules = new Set()
  for file in changedFiles:
    module = resolveModule(file)
    affectedModules.add(module)

  # 4. 只重建受影响的分片
  for module in affectedModules:
    rebuildModuleCard(module)      # modules/{module}.md
    rebuildFileCards(module)       # files/files-{module}.md
    rebuildSymbols(module)         # symbols/ 中该模块的条目

  # 5. 重建聚合索引（如果涉及路由/模型变更）
  if hasRouteChanges(changedFiles):
    rebuildApiRoutes()
  if hasModelChanges(changedFiles):
    rebuildDataModels()

  # 6. 更新元信息
  updateManifest()
  writeLatestDelta(changedFiles, affectedModules)
  updateState(currentHash)
```

### 7.1 集成架构图

```
┌──────────────────────────────────────────────────────────────┐
│                     索引构建阶段                               │
│                                                                │
```

## 8. 与 Skill 系统的关系

### 8.1 对比表

| 维度 | Skill | .ai-index |
|------|-------|-----------|
| **本质** | 开发规范和模式知识 | 项目代码结构数据 |
| **内容** | 怎么写代码（do/don't） | 代码在哪、长什么样 |
| **生命周期** | 手动编写，很少变 | 自动生成，频繁更新 |
| **更新频率** | 周/月级 | 天/次级 |
| **存储位置** | `.claude/skills/` | `.ai-index/` |
| **加载时机** | Hook 评估匹配后激活 | 任务开始前预读 MANIFEST |
| **维护方式** | 人工编写+审核 | 脚本自动生成 |
| **版本管理** | 随项目提交 | 可选提交或 .gitignore |
| **Token 消耗** | 按需加载单个 Skill | 分层加载，渐进式 |

### 8.2 协同工作流

```
用户提任务
  │
  ├─ Hook 评估 → 匹配 Skill（如 crud-development, database-ops）
  │   └─ Opus 读 SKILL.md → 获得开发规范
  │
  ├─ Opus 读 MANIFEST.md → 获得项目全局视图
  │   └─ 读 modules/{module}.md → 获得目标模块上下文
  │
  └─ 两者结合 → Opus 既知道"怎么写"，又知道"改哪里"
      └─ 开始编码
```

### 8.3 不会取代的部分

索引**不会取代** Skill 的以下能力：
- 代码禁止项和规范（code-patterns）
- 框架使用模式（crud-development）
- 错误处理规范（error-handler）
- 安全规范（security-guard）

这些是"怎么写"的知识，不是"代码在哪"的数据。

## 11. 体积控制策略

### 11.1 分层 Token 预算

| 层级 | 单文件上限 | 总量预估（中型项目） | 加载策略 |
|------|----------|-------------------|---------|
| L0 MANIFEST | 500 tokens | 500 tokens | 始终加载 |
| L1 模块卡片 | 1000 tokens/个 | 5000 tokens（5 模块） | 按需 1-2 个 |
| L2 能力索引 | 1500 tokens/个 | 6000 tokens（4 类） | 按需 1-2 个 |
| L3 符号索引 | 1500 tokens/个 | 4500 tokens（3 分片） | 精确查找时 |
| L4 文件卡片 | 1500 tokens/个 | 7500 tokens（5 模块） | 深度了解时 |
| **总量** | - | **~24000 tokens** | **单次加载 1500-3500** |

### 11.2 超标处理

```
当单个分片超过 1500 tokens 时：
  1. 模块卡片超标 → 拆分为 {module}-core.md + {module}-ext.md
  2. API 路由超标 → 按端拆分 api-routes-web.md / api-routes-mobile.md
  3. 符号索引超标 → 增加分片数（A-C, D-F, ...）
  4. 文件卡片超标 → 按子模块拆分

热度降采样：
  - 最近 30 天修改的文件 → 完整摘要（一句话 + 导出符号）
  - 30-90 天未修改 → 精简摘要（只保留路径 + 类型）
  - 90 天以上未修改 → 极简（只保留路径）
```

## 13. 技术决策记录

### TDR-001: 为什么用 Markdown 而不是 JSON/SQLite？

**决策**: 索引文件使用 Markdown 格式

**理由**:
1. AI 用 `Read` 工具直接读取，无需解析器
2. 人类可读，方便调试和审查
3. Git 友好（文本 diff 有意义）
4. 与 Skill 系统一致（都是 .md）

**取舍**: 查询性能不如 SQLite，但 AI 场景是"读取消费"而非"SQL 查询"

### TDR-002: 为什么 MVP 不用 AST？

**决策**: MVP 阶段使用 Grep + 正则提取

**理由**:
1. 零外部依赖（不需要 tree-sitter native 模块）
2. 跨语言通用（正则适配器比 AST parser 简单）
3. 足够满足 80% 场景（类定义、路由、表名）
4. 后续可平滑升级到 AST（适配器接口不变）

**风险**: 正则可能误匹配（注释中的代码、字符串中的 class 等）
**缓解**: 文件卡片标注"置信度"，低置信度条目标记 `~`

### TDR-003: 索引文件是否提交到 Git？

**决策**: 默认 `.gitignore`，可选提交

**理由**:
1. 索引可快速重建（几秒），无需持久化
2. 避免大量自动生成的文件污染 Git 历史
3. 不同开发者的索引可能因分支不同而不同

**例外**: 如果团队希望共享索引（节省每人构建时间），可以提交

### TDR-004: 为什么分层加载而不是一次全加载？

**决策**: L0 → L1 → L2 → L3 → L4 渐进式加载

**理由**:
1. 中型项目全量索引约 24000 tokens，一次加载浪费
2. 多数任务只需 L0 + 1-2 个 L1（约 1500-2500 tokens）
3. 符合"先粗后细"的 AI 工作模式
4. 避免上下文窗口浪费在无关模块信息上
