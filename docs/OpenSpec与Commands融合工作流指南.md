# OpenSpec 与 Commands 融合工作流指南

> OpenSpec 负责**思考**（需求拆解、设计决策），Commands 负责**执行**（代码生成、质量管控）。
> 简单任务跳过 OpenSpec，直接用 Commands；复杂任务先拆解再逐步实现。

## 各节点详细说明

### ① 复杂度快速评估（/dev 第零步）

> 在 `/dev` 命令执行的第一步，AI 自动评估功能复杂度，30 秒内完成。

**评估维度**：

| 维度 | 轻量 | 中等 | 复杂 |
|------|------|------|------|
| 表数量 | 1 张 | 2-3 张 | 4+ 张 |
| 跨模块 | 单模块内 | 偶尔跨模块 | 多模块协作 |
| 业务流程 | 简单 CRUD | 状态流转 | 多步骤审批/复杂流程 |
| 接口数量 | ≤5 个 | 6-10 个 | >10 个 |

**评估结果处理**：

- **轻量/中等** → 直接进入 `/dev` 标准六步流程
- **复杂** → 提示用户考虑先用 OpenSpec 拆解，但用户可回复"继续"跳过

**关键原则**：这是**建议而非强制**，最终决策权在用户手中。

---

### ② OpenSpec 需求拆解

> 当功能复杂时，使用 OpenSpec 将模糊需求拆解为结构化的制品（artifacts）。

**两种拆解方式**：

| 方式 | 命令 | 适用场景 | 过程 |
|------|------|---------|------|
| 快速拆解 | `/opsx:ff [名称]` | 需求已比较清晰 | 一键生成全部制品 |
| 交互式拆解 | `/opsx:new [名称]` | 需求模糊，需讨论 | 逐步创建，每步可调整 |

**制品生成顺序**：

```
┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│ proposal │────▶│  specs/  │────▶│  design  │────▶│  tasks   │
│   .md    │     │  *.md    │     │   .md    │     │   .md    │
└──────────┘     └──────────┘     └──────────┘     └──────────┘
  做什么？          详细规格         怎么实现？        分几步做？
  为什么做？        每个能力的        表结构设计        可执行的
  范围界定          输入/输出         接口清单          开发任务
  成功标准          边界条件          架构决策          清单
```

**各制品详细说明**：

| 制品 | 回答的核心问题 | 内容示例 |
|------|--------------|---------|
| `proposal.md` | 做什么？为什么？ | "开发多级审批流程，支持食堂采购单的三级审批" |
| `specs/*.md` | 每个能力的详细规格 | 输入参数、输出格式、边界条件、异常场景 |
| `design.md` | 技术上怎么实现？ | 表结构设计、接口清单、架构决策、数据流图 |
| `tasks.md` | 分几步完成？ | `[ ] 1. 建表` `[ ] 2. Entity/Mapper` `[ ] 3. Service` ... |

**产出位置**：`openspec/changes/<变更名>/`

### ③ 按任务逐个实现

> 将 OpenSpec 拆解出的任务，逐个通过 Commands 实现。

**两种实现方式**：

| 方式 | 命令 | 过程 |
|------|------|------|
| 自动实现 | `/opsx:apply [name]` | 自动读取 `tasks.md` 中下一个 `[ ]` 未完成任务，结合 `design.md` 技术方案执行 |
| 手动实现 | `/dev [任务描述]` | 用户指定某个具体任务，进入 `/dev` 标准流程 |

**实现循环**：

```
读取 tasks.md 下一个 [ ] 任务
        │
        ▼
  进入 /dev 流程执行
        │
        ▼
  完成后标记 [x]
        │
        ▼
  还有 [ ] 任务？ ──是──▶ 继续下一个
        │
       否
        │
        ▼
  全部完成，进入验证
```

### ④ /dev 代码生成

> 这是核心执行引擎，无论是否经过 OpenSpec 拆解，最终都通过 `/dev` 生成代码。

**六步标准流程**：

```
第一步          第二步          第三步          第四步          第五步          第六步
需求确认   ──▶  重复检查   ──▶  表设计    ──▶  方案确认  ──▶  代码生成  ──▶  完成报告
                                                              │
功能名称        Grep 搜索       建表 SQL        用户确认         8 个文件        生成清单
所属模块        已有 Controller  审计字段        后开始           四层架构        接口文档
端类型          已有 Service     雪花ID                                         下一步建议
包名/路由       ├ 已存在→停止    无 tenant_id
                └ 不存在→继续
```

**每个功能产出的 8 个文件（四层架构）**：

| 层 | 文件 | 路径 | 说明 |
|----|------|------|------|
| Entity | `Xxx.java` | `common/model/` | 数据模型 |
| DTO | `XxxDTO.java` | `web/dto/` | 请求参数 |
| VO | `XxxVO.java` | `web/vo/` | 返回对象 |
| Mapper | `XxxMapper.java` | `common/mapper/` | Mapper 接口 |
| XML | `XxxMapper.xml` | `common/mapper/` | SQL 映射（同目录） |
| Service | `XxxService.java` | `common/service/impl/` | 单表 CRUD |
| Business | `XxxWebBusiness.java` | `web/business/impl/` | 业务编排 |
| Controller | `XxxWebController.java` | `web/controller/` | 接口层 |

### ⑤ 质量管控 & 进度跟踪

> 三个 Commands 持续监控代码质量和开发进度，融合后新增了 OpenSpec 变更状态的感知。

#### /check - 代码规范检查

10 项检查，不涉及 OpenSpec：

| 检查项 | 说明 |
|--------|------|
| 包名规范 | 必须 `net.xnzn.core.*` |
| 审计字段 | `crby/crtime/upby/uptime`（非 createBy/createTime） |
| del_flag 语义 | `2=正常，1=删除`（非 0=正常） |
| 权限注解 | 所有接口必须有 `@RequiresAuthentication` 或 `@RequiresGuest` |
| 请求封装 | POST 请求体使用 `LeRequest<T>` |
| 对象转换 | 使用 `BeanUtil.copyProperties()`（非 MapstructUtils） |
| 异常抛出 | 使用 `LeException`（非 ServiceException） |
| 无 tenant_id | 双库物理隔离，Entity 不加 tenant_id |
| 验证包 | `jakarta.validation.*`（非 javax） |
| Mapper XML 位置 | 与 Mapper.java 同目录 |

#### /progress - 进度报告（增强）

| 扫描内容 | 原有 | 融合后新增 |
|---------|------|-----------|
| 模块完整性 | 4 模块 × 8 文件检查 | - |
| 代码待办 | TODO/FIXME 统计 | - |
| 代码质量 | 规范性检查 | - |
| **OpenSpec 状态** | - | 扫描 `openspec/changes/*/tasks.md`，展示变更名/状态/任务完成率 |

输出示例：

```markdown
## OpenSpec 变更状态

| 变更名称 | 状态 | 任务完成 | 概述 |
|---------|------|---------|------|
| purchase-approval | 进行中 | 3/7 | 采购单多级审批流程 |
| meal-booking | 进行中 | 0/4 | 餐食预订功能 |
```

#### /next - 下一步建议（增强）

| 分析内容 | 原有 | 融合后新增 |
|---------|------|-----------|
| Git 提交分析 | 最近提交、频率 | - |
| TODO/FIXME | 代码待办统计 | - |
| 规范问题 | 包名、工具类检查 | - |
| **OpenSpec 变更** | - | 扫描活跃变更，将未完成任务纳入建议优先级 |

新增建议示例：

```markdown
### 如果有未完成的 OpenSpec 变更：

**推荐：继续实现已拆解的任务**
- 变更名称：purchase-approval
- 未完成任务：4. 审批 Service 层实现、5. 审批 Business 编排...
- 快速开始：/opsx:apply purchase-approval
```

### ⑥ 变更完成 & 归档

> 仅走过 OpenSpec 流程的功能需要这一步。

| 操作 | 命令 | 作用 |
|------|------|------|
| 验证 | `/opsx:verify [name]` | 对比 `tasks.md` 和实际代码，确认所有任务已正确实现 |
| 归档 | `/opsx:archive [name]` | 将变更目录移入 `openspec/archive/`，保留历史记录 |
| 批量归档 | `/opsx:bulk-archive` | 一次性归档多个已完成的变更 |

## 两种典型使用路径

### 路径 A：简单功能（轻量/中等）

适用于单表 CRUD、字段增删、简单查询等场景。

```
用户: /dev 员工考勤管理

  ① 评估: 1张表、单模块、简单CRUD、3个接口 → 轻量
  ② 直接走 /dev 六步流程
     → 需求确认 → 重复检查 → 表设计 → 方案确认 → 代码生成 → 完成
  ③ /check 验证规范
  ④ 完成
```

**耗时**：一次对话完成，无额外流程开销。

### 路径 B：复杂功能

适用于多表联动、跨模块协作、复杂业务流程等场景。

```
用户: /dev 多级采购审批流程

  ① 评估: 4张表、跨 canteen+drp 模块、审批流程 → 复杂
  ② 建议 OpenSpec，用户同意
  ③ /opsx:ff purchase-approval
     → proposal.md（做什么）
     → specs/approval-flow.md（审批规格）
     → specs/notification.md（通知规格）
     → design.md（表结构、接口设计）
     → tasks.md（7个任务）
  ④ 逐个实现
     → /opsx:apply purchase-approval（任务1: 建表）
     → /opsx:apply purchase-approval（任务2: Entity/Mapper）
     → /dev 审批Service（任务3: 手动选择）
     → ...
  ⑤ /progress 查看进度（含 OpenSpec 状态: 5/7 完成）
  ⑥ /next 查看建议（提示: 还有2个未完成任务）
  ⑦ 完成全部任务
  ⑧ /opsx:verify purchase-approval
  ⑨ /opsx:archive purchase-approval
```

**耗时**：多次对话，但每步目标明确，不会遗漏。

## 开发模式选择指南

### 按复杂度选择

| 复杂度 | 判断标准 | 推荐工作流 | 说明 |
|--------|---------|-----------|------|
| 轻量 | 单表 CRUD、字段增删、简单查询 | `/dev` 或 `/crud` 直接开发 | 无需需求拆解 |
| 中等 | 2-3 张表联动、跨 Service 调用、状态流转 | `/dev` 开发（可选 OpenSpec） | 建议先理清逻辑再编码 |
| 复杂 | 多模块协作、复杂业务流程、需要设计评审 | OpenSpec → `/dev` | 先拆解再逐步实现 |

### 按开发类型选择

| 类型 | 特征 | 工作流 |
|------|------|--------|
| 产品开发（core 迭代） | 改 core 代码、会有后续迭代 | 复杂功能用 OpenSpec 拆解，简单功能直接 `/dev` |
| 项目定制（一次性） | Dz 前缀类、dz_ 表、@Primary 覆盖 | 直接 `/dev`，参考 `leniu-customization-location` 技能 |

## 命令速查表

### Commands（执行类）

| 命令 | 用途 | 何时使用 |
|------|------|---------|
| `/dev` | 开发新功能（含表设计 + 代码生成） | 从零开发新功能 |
| `/crud` | 快速生成 CRUD 代码 | 表已存在时 |
| `/check` | 10 项代码规范检查 | 开发完成后、定期检查 |
| `/progress` | 项目进度报告 + OpenSpec 状态 | 了解整体完成情况 |
| `/next` | 下一步建议 + OpenSpec 未完成任务 | 不知道接下来做什么 |
| `/start` | 快速了解项目 | 初次接触项目 |
| `/add-todo` | 添加待办任务 | 记录待办事项 |

### OpenSpec（思考类）

| 命令 | 用途 | 何时使用 |
|------|------|---------|
| `/opsx:ff [名称]` | 一键生成全部制品 | 需求已清晰，快速拆解 |
| `/opsx:new [名称]` | 交互式创建变更 | 需求模糊，需要讨论 |
| `/opsx:apply [名称]` | 实现下一个任务 | 按任务顺序开发 |

| `/opsx:archive [名称]` | 归档已完成的变更 | 验证通过后 |

## OpenSpec 与 Commands 的分工

```
OpenSpec（思考层）                Commands（执行层）
─────────────────                ─────────────────

proposal.md    ─── 需求驱动 ───▶  /dev（按 task 逐个开发）
  "做什么"                         "生成代码"

specs/*.md     ─── 规格指导 ───▶  /crud（快速生成 CRUD）
  "详细需求"                       "表已有时快速生成"

design.md      ─── 设计约束 ───▶  /check（验证规范）
  "怎么实现"                       "是否符合规范"

tasks.md       ─── 进度源 ────▶  /progress（跟踪进度）
  "做了多少"                       "整体完成率"
                                  /next（下一步建议）
                                   "接下来做什么"
```

**核心原则**：
- OpenSpec 产出的是**文档**（.md 制品），不是代码
- Commands 产出的是**代码**（.java 文件），不是文档
- 两者通过 `tasks.md` 连接：OpenSpec 写任务，Commands 执行任务
